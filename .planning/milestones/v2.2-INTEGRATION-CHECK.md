## Changes Since Last Check

### New since prior audit (in scope of this re-audit)

1. **Phase 22.1 (RETR-02/03 wiring closure)** — added `summaryToSearchResult` export in `src/pensieve/routing.ts` and updated all five chat-mode handlers to route through `retrieveContext` with their per-mode `*_SEARCH_OPTIONS` passed through the new `hybridOptions` parameter. This closes the orphan-orchestrator finding in the prior check.
2. **Phase 21 WR-01 fix** — `buildEntriesBlock` (src/episodic/prompts.ts:278) now renders entry HH:MM timestamps in `input.tz` via `DateTime.fromJSDate(e.createdAt, { zone: tz })`, not UTC. The `tz` field was added to `ConsolidationPromptInput` and is threaded from `config.proactiveTimezone` through `runConsolidate(date)` → `assembleConsolidationPrompt({ ..., tz })` → `buildEntriesBlock(entries, summaryDate, tz)`.
3. **Phase 22 WR-02 fix** — all three regex fast-paths in `src/chris/modes/date-extraction.ts` now anchor at T12:00:00Z (noon UTC): `matchIsoDate` at line 64, `matchRelativeAgo` via `setUTCHours(12, 0, 0, 0)` at line 144, `matchMonthDay` at line 211 / 218, and the Haiku fallback at line 290. This eliminates the negative-offset tz drift where a midnight-UTC Date formatted back in America/* tz resolved to the prior calendar day.
4. **Phase 22 WR-04 fix** — `matchMonthDay` now rejects calendar-invalid dates (Feb 30, April 31, Feb 29 in non-leap years) via `d.getUTCMonth() !== monthIdx || d.getUTCDate() !== day` at line 224. Previously `Date.UTC` silently rolled Feb 30 → March 2.
5. **Phase 22 WR-03 fix** — FR and RU branches of `matchMonthDay` now use leading `\b` to prevent "21 décembre" from being extracted out of "item 121 décembre".
6. **Phase 23 WR-01 fix** — `handleSummaryCommand` (src/bot/handlers/summary.ts:179-183) now round-trips the ISO date argument through `DateTime.fromISO(after, { zone: 'utc' }).isValid` before accepting it. Inputs like `2026-02-30` / `2026-13-01` / `2026-04-31` that previously passed the regex-only check now receive the localized usage help.
7. **Phase 20 WR-02 fix** — `regen-snapshots.sh` EXIT trap now guards deletion of `0005_snapshot.json` behind the `REGEN_PRODUCED_0005=1` flag (set at line 324, just before the acceptance-gate generate). A re-run of the script after Plan 20-02 landed a real 0005 snapshot no longer destroys the committed file.
