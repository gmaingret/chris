## Warnings

### WR-01: `beforeAll` existence probe is effectively tautological

**File:** `src/episodic/__tests__/schema.test.ts:38-43`
**Issue:** The probe

```ts
const result = await sql`SELECT 1 as ok FROM episodic_summaries LIMIT 0`;
expect(Array.isArray(result)).toBe(true);
```

is a weak proof that migration 0005 applied. `LIMIT 0` always returns an empty array when
the query succeeds, and the `Array.isArray` check would also be `true` for any successful
query (postgres-js returns arrays on success). The assertion itself adds nothing beyond
"the query didn't throw" — and if it did throw, Vitest would surface the error anyway.
Worse, if a future postgres-js upgrade returns a different shape on `LIMIT 0` (e.g. a
command-tag object), the assertion silently breaks the existence proof without changing
test behavior.

**Fix:** Either drop the assertion and let the thrown-error path be the proof, or tighten
it to verify the table specifically:

```ts
await sql`SELECT 1 FROM episodic_summaries LIMIT 0`; // throws if table missing → test fails
// ...or, for a positive assertion:
const rows = await sql<{ exists: boolean }[]>`
  SELECT EXISTS (
    SELECT 1 FROM information_schema.tables
    WHERE table_name = 'episodic_summaries'
  ) AS exists
`;
expect(rows[0]?.exists).toBe(true);
```

### WR-02: `set -e` + `trap EXIT` `cleanup()` can swallow original exit codes

**File:** `scripts/regen-snapshots.sh:78-89`
**Issue:** Inside the EXIT trap:

```bash
cleanup() {
  local rc=$?
  echo ""
  echo "🧹 Cleaning up regen postgres..."
  docker compose ... down --volumes --timeout 5 >/dev/null 2>&1 || true
  rm -rf "${TMP_DIR}/drizzle-regen-"* "${OUT_DIR}" "${OVERRIDE_FILE}" || true
  find "${MIGRATIONS_DIR}" -maxdepth 1 -name "0005_acceptance_check*.sql" -delete 2>/dev/null || true
  find "${META_DIR}" -name "0005_snapshot.json" -delete 2>/dev/null || true
  exit "${rc}"
}
```

The `find ... -delete 2>/dev/null || true` chain silently deletes
`${META_DIR}/0005_snapshot.json` on *every* exit, including successful ones. Since
this script is intended to be re-runnable and Phase 20-02 writes a real
`0005_snapshot.json` (after this script lands in Phase 20-01), running this script
against an already-migrated tree would delete the committed snapshot. The
`acceptance_check` naming convention isn't grepped — any `0005_snapshot.json` is
purged.

This is not currently exploitable (Plan 20-01 committed the script before Plan 20-02
added the 0005 snapshot), but the script is documented as "re-runnable for any future
snapshot recovery". A user following that guidance after Phase 20-02 would silently
destroy their committed 0005 snapshot.

**Fix:** Scope the cleanup to paths the script *actually created* during this run:

```bash
cleanup() {
  local rc=$?
  echo "🧹 Cleaning up regen postgres..."
  docker compose -p "${COMPOSE_PROJECT}" ... down --volumes --timeout 5 >/dev/null 2>&1 || true
  rm -rf "${TMP_DIR}/drizzle-regen-"* "${OUT_DIR}" "${OVERRIDE_FILE}" || true
  # Only delete 0005 snapshot if the acceptance-check name was produced by THIS run
  find "${MIGRATIONS_DIR}" -maxdepth 1 -name "0005_acceptance_check*.sql" -delete 2>/dev/null || true
  # Do NOT blanket-delete 0005_snapshot.json — it may be a legitimate committed snapshot
  # (Phase 20-02 ships one). The name `0005_acceptance_check` above is distinctive;
  # drizzle-kit may also write a generic `0005_snapshot.json` — key off the paired SQL
  # file's existence instead:
  if compgen -G "${MIGRATIONS_DIR}/0005_acceptance_check*.sql" >/dev/null; then
    find "${META_DIR}" -name "0005_snapshot.json" -newer "${TMP_DIR}" -delete 2>/dev/null || true
  fi
  exit "${rc}"
}
```

Or simpler: snapshot the pre-run mtime of `${META_DIR}/0005_snapshot.json` if present, and
only delete if mtime changed.

### WR-03: `introspect_to()` loses diagnostic output on drizzle-kit failure

**File:** `scripts/regen-snapshots.sh:147-160`
**Issue:**

```bash
(
  cd "${work_dir}"
  yes '' 2>/dev/null | npx drizzle-kit introspect >&2 2>&1 || true
) >&2
```

The `|| true` swallows drizzle-kit's non-zero exit code, then the next line's file-not-
found check emits a generic error. If introspect fails for a reason other than "no
snapshot emitted" (e.g. DB connection timeout, schema parse error), the script logs
"introspect did not produce a snapshot" without the actual error context. Debugging is
harder than necessary.

**Fix:** Capture the exit code explicitly and surface it:

```bash
(
  cd "${work_dir}"
  yes '' 2>/dev/null | npx drizzle-kit introspect >&2 2>&1
) >&2
local rc=$?
if [[ "${rc}" -ne 0 ]]; then
  echo "❌ drizzle-kit introspect failed with exit ${rc}" >&2
  return "${rc}"
fi
```

## Info

### IN-01: `.notNull().defaultNow()` order inconsistent with codebase convention

**File:** `src/db/schema.ts:330`
**Issue:** The new `episodicSummaries.createdAt` uses `.notNull().defaultNow()`, while
every other `created_at` in `schema.ts` uses `.defaultNow().notNull()` (12 occurrences at
lines 107, 126, 140, 152, 168, 187, 244, 263, 283, 297 — the one outlier is the new Phase 20 row).
Functionally identical; the drizzle chain is commutative here. Minor style drift.

**Fix:** Swap method order for consistency with surrounding code:

```ts
createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
```

### IN-02: `source_entry_ids` Zod schema lacks `.min(1)` despite "non-empty in practice" comment

**File:** `src/episodic/types.ts:37`
**Issue:** The schema is

```ts
source_entry_ids: z.array(z.string().uuid()),
```

The comment immediately above says "non-empty array of uuid strings in practice (CONS-02
skips zero-entry days)". The "in practice" wording is intentional — Phase 21's
`runConsolidate()` gates this upstream via an entry-count check — but the Zod schema
itself still accepts `[]`. This is a defense-in-depth gap: an OPS-01 backfill run or a
direct operator call to `parseEpisodicSummary()` could insert a zero-source row, which
then fails soft (no error, but the audit trail loses the "which entries informed this"
link).

If the intent is "Zod accepts []; upstream enforces non-empty", the current state is
correct and documented. If you want Zod to be the second line of defense (matching the
DB-level CHECK-on-importance philosophy from D-07), add `.min(1)`:

```ts
source_entry_ids: z.array(z.string().uuid()).min(1),
```

No action required if the current intent is "upstream-only enforcement" — flagging for
explicit-decision confirmation.

### IN-03: `PgLikeError` type duplicated across two `it()` blocks

**File:** `src/episodic/__tests__/schema.test.ts:116-121` and `149-154`
**Issue:** The same `PgLikeError` type alias is declared inside both the UNIQUE and CHECK
test bodies. Identical 6-line blocks. If the shape changes (e.g. postgres.js adds a
`severity` field that you want to assert on), you'd edit two places.

**Fix:** Hoist to module scope:

```ts
// ── Helpers ─────────────────────────────────────────────────────────────────
type PgLikeError = {
  code?: string;
  message?: string;
  constraint_name?: string;
  cause?: { code?: string; message?: string; constraint_name?: string };
};
```

### IN-04: `beforeEach` deletes without a WHERE clause

**File:** `src/episodic/__tests__/schema.test.ts:48`
**Issue:** `await db.delete(episodicSummaries)` truncates the whole table. Safe today
(the episodic test is the only consumer of this table in the test run, and the data is
local-only), but future tests in the same file that want to seed and verify across
`beforeEach` boundaries would also be wiped. If the table grows FK references from
another test's fixtures, this silently becomes a cascade-delete surprise.

**Fix:** Not strictly required today. If adopted for defensive isolation:

```ts
beforeEach(async () => {
  await sql`TRUNCATE episodic_summaries RESTART IDENTITY CASCADE`;
});
```

Or keep as-is with an inline comment justifying the unscoped delete (the current inline
comment at line 46-47 covers it reasonably — this is a very soft nit).

### IN-05: `--check-only` is the only recognized flag; unknown args silently accepted

**File:** `scripts/regen-snapshots.sh:58-61`
**Issue:**

```bash
CHECK_ONLY=0
if [[ "${1:-}" == "--check-only" ]]; then
  CHECK_ONLY=1
fi
```

Any other first-arg value (typo like `--check_only`, a stray flag like `--help`, a
misrouted path) is silently treated as "full run". No usage help is printed for invalid
input.

**Fix:** Tighten argument handling:

```bash
CHECK_ONLY=0
case "${1:-}" in
  "")                ;;
  "--check-only")    CHECK_ONLY=1 ;;
  "--help"|"-h")
    sed -n '2,40p' "$0"  # print the top-of-file docstring
    exit 0
    ;;
  *)
    echo "Unknown argument: ${1}" >&2
    echo "Usage: $0 [--check-only|--help]" >&2
    exit 2
    ;;
esac
```

### IN-06: `node -e` with shell-interpolated paths is theoretically injectable

**File:** `scripts/regen-snapshots.sh:169-176, 191-200, 182, 205`
**Issue:** The `patch_snapshot_chain`, `patch_prev_id_inplace`, `snapshot_id`, and
`new_uuid` helpers build node programs via bash heredocs:

```bash
node -e "
  const fs = require('fs');
  const s = JSON.parse(fs.readFileSync('${src}', 'utf8'));
  s.id = '${new_id}';
  s.prevId = '${prev_id}';
  fs.writeFileSync('${dst}', JSON.stringify(s, null, 2));
"
```

If any of `${src}`, `${dst}`, `${new_id}`, `${prev_id}` contained a single quote, newline,
or `${...}` shell expansion, the resulting JS would be malformed or execute unintended
code. In practice all inputs are script-controlled (hard-coded paths, `crypto.randomUUID()`
output, values read from internal JSON snapshot ids) — so the current surface is benign.
But this pattern is a footgun if the script is ever extended to accept user-supplied
paths or UUIDs.

**Fix (defensive):** Pass values via env vars, which node reads safely:

```bash
patch_snapshot_chain() {
  local src="$1" dst="$2" new_id="$3" prev_id="$4"
  SRC="$src" DST="$dst" NEW_ID="$new_id" PREV_ID="$prev_id" node -e '
    const fs = require("fs");
    const s = JSON.parse(fs.readFileSync(process.env.SRC, "utf8"));
    s.id = process.env.NEW_ID;
    s.prevId = process.env.PREV_ID;
    fs.writeFileSync(process.env.DST, JSON.stringify(s, null, 2));
  '
}
```

No urgency — inputs are currently sanitized-by-construction. Flagging as a hardening
opportunity for a script that may be extended later.
