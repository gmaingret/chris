# Phase 22.1: Code Review Report

**Reviewed:** 2026-04-18
**Depth:** standard
**Files Reviewed:** 12
**Status:** findings

## Summary

Phase 22.1 wires `retrieveContext` into the five chat-mode handlers (JOURNAL / REFLECT / COACH / PSYCHOLOGY / PRODUCE) via a new `hybridOptions?: SearchOptions` passthrough and a `summaryToSearchResult` adapter. The change is a clean, localized wiring refactor — the per-mode identity (tags / recencyBias / minScore / limit) is preserved round-trip, INTERROGATE and `/summary` are byte-identical per their documented bypass, and each mode has three new regression tests covering the canonical routing decisions (recent→raw, old→summary, verbatim→raw).

The code is correct for all 4 routing branches (verbatim / recent / no-summary-fallback / catch-block fallback) and the merge semantics (`hybridOptions.limit ?? rawLimit`) are well-chosen. The `summaryToSearchResult` adapter correctly synthesizes a typed SearchResult with score=1.0 sentinel that survives the 0.3 threshold, uses the `episodic-` id namespace to prevent UUID collision, and renders via the existing `buildPensieveContext` pipeline.

Findings are all non-blocking quality issues. The single Warning (WR-01) flags a missed test-coverage opportunity: `summaryToSearchResult` and the `hybridOptions` passthrough have zero direct unit tests in `routing.test.ts` — they are only exercised transitively through the 5 mode test files. The Info items cover minor cleanup (redundant `rawLimit` argument at call sites, commented-code inconsistency between test files, one comment drift in journal.ts about date suppression, the 5× duplication in mode handlers explicitly allowed by project rules, and a subtle system-prompt shape that lets the summary date appear in JOURNAL output despite `includeDate: false`).

No security issues, no logic bugs, no regressions against the Phase 22 retrieveContext contract. Zero changes to INTERROGATE and `/summary` confirmed by the project's own git-diff verification.

## Warnings

### WR-01: No direct unit tests for `summaryToSearchResult` or `hybridOptions` passthrough in `routing.test.ts`

**File:** `src/pensieve/routing.ts:137-280` (subjects) / `src/pensieve/__tests__/routing.test.ts` (missing coverage)
**Issue:** The Phase 22.1 SUMMARY states "22 existing routing.test.ts tests pass byte-identically" — which is true, but the existing tests were written pre-22.1 and cover the routing decision tree only. The two new exports (`summaryToSearchResult`, and the `hybridOptions` merge behavior inside `retrieveContext`) are tested only transitively through the 5 mode test files. `grep summaryToSearchResult src/pensieve/__tests__/routing.test.ts` returns empty; `grep hybridOptions src/pensieve/__tests__/routing.test.ts` returns empty.

This matters for three reasons:
1. **Contract tests for `summaryToSearchResult` are load-bearing** — the header format `[Episode Summary YYYY-MM-DD | importance=N/10 | topics=...]`, the score=1.0 sentinel, the `episodic-${summary.id}` namespace prefix, the UTC-midnight `createdAt` derivation, and the empty-topics fallback (`'none'`) are all independent guarantees that the mode-test assertions only spot-check.
2. **The `hybridOptions.limit ?? rawLimit` precedence rule** — the merge semantics (mode-specific `.limit` wins, routing-level `rawLimit` as fallback, empty `hybridOptions` defaults to `rawLimit`) have zero routing-layer regression protection. A future contributor could invert the precedence and the mode tests would still pass (they all pass a `rawLimit` equal to `hybridOptions.limit`, so the inversion is invisible at the mode layer).
3. **The 4-call-site merge propagation** — routing.ts line 151 / 166 / 184 / 232 all must pass `mergedHybridOptions` to `hybridSearch`. If someone accidentally reverts one of those to `{ limit: rawLimit }`, only an integration test that sends a mode-specific tag filter through an old-ish query on the no-summary-fallback branch would catch it — and no such test exists.

**Fix:** Add a unit-test block to `src/pensieve/__tests__/routing.test.ts`:

```typescript
describe('summaryToSearchResult', () => {
  const baseSummary = {
    id: '11111111-1111-1111-1111-111111111111',
    summaryDate: '2026-03-15',
    summary: 'Greg reflected on relocation',
    importance: 5,
    topics: ['relocation', 'batumi'],
    emotionalArc: 'reflective',
    keyQuotes: [],
    sourceEntryIds: [],
    createdAt: new Date('2026-03-15T12:00:00Z'),
  };

  it('wraps the summary with score=1.0 sentinel', () => {
    const result = summaryToSearchResult(baseSummary);
    expect(result.score).toBe(1.0);
  });

  it('prepends labeled inline header to content', () => {
    const result = summaryToSearchResult(baseSummary);
    expect(result.entry.content).toContain(
      '[Episode Summary 2026-03-15 | importance=5/10 | topics=relocation, batumi]',
    );
    expect(result.entry.content).toContain('Greg reflected on relocation');
  });

  it('renders topics=none when the summary has no topics', () => {
    const result = summaryToSearchResult({ ...baseSummary, topics: [] });
    expect(result.entry.content).toContain('topics=none');
  });

  it('namespaces the synthetic id with episodic- prefix (spoofing mitigation)', () => {
    const result = summaryToSearchResult(baseSummary);
    expect(result.entry.id).toBe('episodic-11111111-1111-1111-1111-111111111111');
  });

  it('sets createdAt to UTC midnight of summaryDate', () => {
    const result = summaryToSearchResult(baseSummary);
    expect(result.entry.createdAt?.toISOString()).toBe('2026-03-15T00:00:00.000Z');
  });
});

describe('retrieveContext hybridOptions passthrough', () => {
  // Using the existing mock-hybridSearch scaffolding in routing.test.ts:
  it('merges hybridOptions into every hybridSearch call (verbatim branch)', async () => {
    const opts = { tags: ['BELIEF'] as const, recencyBias: 0.5, limit: 7 };
    await retrieveContext({ query: 'what did I say exactly', hybridOptions: opts });
    expect(mockHybridSearch).toHaveBeenCalledWith(
      'what did I say exactly',
      expect.objectContaining({ tags: ['BELIEF'], recencyBias: 0.5, limit: 7 }),
    );
  });

  it('uses rawLimit when hybridOptions.limit is absent', async () => {
    await retrieveContext({
      query: 'q',
      rawLimit: 13,
      hybridOptions: { recencyBias: 0.5 },  // no limit
    });
    expect(mockHybridSearch).toHaveBeenCalledWith(
      'q',
      expect.objectContaining({ limit: 13, recencyBias: 0.5 }),
    );
  });

  it('lets hybridOptions.limit override rawLimit when both are set', async () => {
    await retrieveContext({
      query: 'q',
      rawLimit: 13,
      hybridOptions: { limit: 7 },
    });
    expect(mockHybridSearch).toHaveBeenCalledWith('q', expect.objectContaining({ limit: 7 }));
  });

  // Repeat for the recent / no-summary-fallback / catch-block branches with
  // 30d-old queryDate + getEpisodicSummary returning null / throwing.
});
```

This cost is small (~40 lines) and locks in the two Phase 22.1 contracts that currently have zero routing-layer regression protection.

## Info

### IN-01: Redundant `rawLimit` argument at all 5 mode handler call sites

**File:** `src/chris/modes/journal.ts:46`, `reflect.ts:42`, `coach.ts:42`, `psychology.ts:43`, `produce.ts:43`
**Issue:** Each handler passes `rawLimit: <MODE>_SEARCH_OPTIONS.limit` and `hybridOptions: <MODE>_SEARCH_OPTIONS` — but since `hybridOptions.limit` wins in the merge (`limit: opts.hybridOptions?.limit ?? rawLimit`), the `rawLimit` argument is always dead code when both are drawn from the same preset constant. It never changes the behavior.

This is not a bug, but it creates two minor hazards:
- A future reader sees the two arguments and has to reason about precedence to realize `rawLimit` is inert.
- If someone later changes `<MODE>_SEARCH_OPTIONS.limit` but leaves `rawLimit: 10` hard-coded (by mistake), the mistake is invisible at the call site (hybridOptions.limit still wins).

**Fix:** Drop `rawLimit` from the 5 handlers. `hybridOptions` carries the limit, and if `hybridOptions.limit` were ever undefined, the routing-layer default (`opts.rawLimit ?? 10`) would kick in:

```typescript
const routing = await retrieveContext({
  query: text,
  queryDate,
  hybridOptions: JOURNAL_SEARCH_OPTIONS,  // .limit is already here
});
```

If you want an explicit "hand-me-the-limit-from-the-preset" invariant at the call site for readability, convert it to a comment rather than a dead argument:

```typescript
// Limit comes from JOURNAL_SEARCH_OPTIONS.limit via hybridOptions
const routing = await retrieveContext({ query: text, queryDate, hybridOptions: JOURNAL_SEARCH_OPTIONS });
```

### IN-02: Inconsistent `vi.mock` factory reference pattern between test files

**File:** `src/chris/__tests__/journal.test.ts:51` vs `reflect.test.ts:48`, `coach.test.ts:46`, `psychology.test.ts:46`, `produce.test.ts:46`
**Issue:** `journal.test.ts` uses the hoist-safe indirect form `hybridSearch: (...args: unknown[]) => mockHybridSearch(...args)`. The other four mode test files use the direct form `hybridSearch: mockHybridSearch`. Both patterns work under vitest's `mock*`-prefix hoisting exemption, but the inconsistency makes copy-paste refactors fragile: a contributor copying journal.test.ts's pattern into a new test would not match the established style in the other four files, and vice versa.

**Fix:** Pick one style across the 5 mode test files. The direct `hybridSearch: mockHybridSearch` form is shorter and is what the other four already use — update journal.test.ts to match:

```typescript
vi.mock('../../pensieve/retrieve.js', () => ({
  hybridSearch: mockHybridSearch,
  getEpisodicSummary: mockGetEpisodicSummary,
  JOURNAL_SEARCH_OPTIONS: { ... },
}));
```

### IN-03: `chris.<mode>.routing` log is emitted before `buildPensieveContext` in all 5 handlers — a log-before-consumer ordering that misses a data point

**File:** `src/chris/modes/journal.ts:52-60`, `reflect.ts:48-56`, `coach.ts:48-56`, `psychology.ts:49-57`, `produce.ts:49-57`
**Issue:** The uniform `chris.<mode>.routing` log fires immediately after `retrieveContext` returns, carrying `{ chatId, reason, hasSummary, rawCount }`. But the ACTUAL payload that reaches the LLM is the `searchResults` array (summary prepended when present) — not just `raw`. A log consumer who wants to answer "how many citations did this mode ship to Sonnet on March 15?" has to compute `rawCount + (hasSummary ? 1 : 0)` by hand.

This is minor observability noise, not a bug. But since the `chris.<mode>.routing` log is marketed as THE diagnostic for the Phase 22.1 wiring, adding one more field removes ambiguity at aggregation time.

**Fix:** Add a `searchResultCount` field (or rename `rawCount` to `citationCount` and fold in the summary):

```typescript
logger.info(
  {
    chatId: chatId.toString(),
    reason: routing.reason,
    hasSummary: routing.summary != null,
    rawCount: routing.raw.length,
    citationCount: searchResults.length,  // post-merge total that Sonnet sees
  },
  'chris.journal.routing',
);
```

Non-breaking addition; all 5 handlers can adopt it in one sweep.

### IN-04: Synthetic-summary content carries a date marker that bypasses JOURNAL's `includeDate: false` suppression

**File:** `src/chris/modes/journal.ts:62` + `src/pensieve/routing.ts:265`
**Issue:** JOURNAL calls `buildPensieveContext(searchResults, { includeDate: false })` specifically to prevent today's seed-stamp from leaking into the rendered citation header (Phase 11 / RETR-04). But when the summary-tier fires, the synthetic SearchResult's content is `[Episode Summary 2026-03-15 | importance=5/10 | topics=...]\n<summary text>` — the date `2026-03-15` is embedded inside the content string itself, not in the citation header. `includeDate: false` only suppresses the `(YYYY-MM-DD | ...)` prefix added by context-builder; it does NOT scrub dates from entry content.

This may be INTENTIONAL — the episode summary date is semantically meaningful (it's the date the user is asking about, not the storage timestamp), so anchoring the summary to that date is probably correct. The SUMMARY's key-decision #1 explicitly calls out the inline form as a deliberate choice. And in fact, an episode-summary citation WITHOUT a date is less useful than one with a date.

**Fix:** If the inline date is intentional (which I believe it is), update the comment on `src/chris/modes/journal.ts:61` to be explicit that the `includeDate: false` invariant applies to RAW entries only, not to synthetic episodic-summary content:

```typescript
// Phase 11 / RETR-04: suppress per-entry date prefix on RAW entries so
// today's seed-stamp does not leak into fabrications. Synthetic episodic-
// summary entries (via summaryToSearchResult) carry their own episode date
// inside content — that date is semantically anchored to the user's query
// and is MEANT to be visible.
const pensieveContext = buildPensieveContext(searchResults, { includeDate: false });
```

Without this comment, a reader who later sees the summary-date in a JOURNAL prompt output may think the suppression is broken.

### IN-05: Five-handler duplication is intentional but the uniform block has drifted between copies

**File:** `src/chris/modes/journal.ts:42-60`, `reflect.ts:38-56`, `coach.ts:38-56`, `psychology.ts:39-57`, `produce.ts:39-57`
**Issue:** Per the project's "5 handlers each need 3 lines is better than premature abstraction" rule (cited in the review focus), the ~10-line `extractQueryDate → retrieveContext → summaryToSearchResult prepend → log` block is deliberately duplicated. That is accepted.

However, the 5 copies have already drifted — not in behavior, but in comment style:
- journal.ts (line 37-41): 4-line docstring mentioning "JOURNAL_SEARCH_OPTIONS (tags filter + recencyBias + limit)"
- reflect.ts (line 35-37): 3-line docstring mentioning "REFLECT_SEARCH_OPTIONS round-trip"
- coach.ts (line 35-37): 3-line docstring mentioning "COACH_SEARCH_OPTIONS (BELIEF/INTENTION/VALUE tag filter + recencyBias 0.5)"
- psychology.ts (line 35-38): 4-line docstring mentioning "PSYCHOLOGY_SEARCH_OPTIONS (EMOTION/FEAR/BELIEF/DREAM tag filter + recencyBias 0.2 + limit 15)"
- produce.ts (line 36-38): 3-line docstring

The content and blast-radius of the mode-specific options is also baked into each comment — which means any future change to `<MODE>_SEARCH_OPTIONS` requires updating the inline comment in the matching handler, and it's easy to miss. This is the classic drift hazard of the "duplicate-by-design" pattern.

**Fix:** Keep the duplication (the project rule is sound), but remove the mode-specific option details from the comments — let the imported constant speak for itself:

```typescript
// Phase 22.1 RETR-02/03: route through retrieveContext so old-dated queries
// escalate to the episodic-summary tier. Mode-specific search options
// round-trip via hybridOptions; see the const definition in retrieve.ts
// for the current tag/recencyBias/limit values.
const queryDate = await extractQueryDate(text, language);
const routing = await retrieveContext({
  query: text,
  queryDate,
  hybridOptions: JOURNAL_SEARCH_OPTIONS,
});
```

This way the 5 copies stay byte-identical modulo the `<MODE>_SEARCH_OPTIONS` symbol, and a future edit to `COACH_SEARCH_OPTIONS` cannot silently desync the comment in coach.ts.
