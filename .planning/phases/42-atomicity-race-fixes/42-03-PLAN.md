<objective>
Fix RACE-06: weekly-review fire pipeline must be transactional with respect to the Telegram send. Today's order sets `respondedAt` and the `pensieve_entry_id` back-reference BEFORE the send, so a transient Grammy failure (`429 Too Many Requests`, network drop, etc.) silently records "fired successfully" in the response row while no Telegram message was actually delivered — a silent weekly miss.

Per Phase 29 BL-01 review + D-42-11 + D-42-12 + D-42-13:
- Pre-allocate `ritual_responses` row WITHOUT `respondedAt`.
- Persist Pensieve entry (Pensieve persistence is allowed to orphan on send-fail — the observation text is useful for forensics and the back-reference resolves on retry).
- Send Telegram inside `try { ... } catch { ... }`.
- **On success:** `db.transaction` runs `ritual_responses` UPDATE (set respondedAt + pensieveEntryId) + `ritual_fire_events` INSERT (outcome='fired') as a single atomic commit.
- **On failure:** `ritual_fire_events` INSERT outcome='fired' + `metadata.telegram_failed: true` for audit visibility; `rituals.next_run_at` reverted to its pre-fire value so next Sunday's sweep retries; `ritual_responses.respondedAt` stays NULL; Pensieve entry remains. Log `'rituals.weekly.send_failed'` at ERROR level.

Output: One production file modified (weekly-review.ts), one test file modified (weekly-review.test.ts adds a new RACE-06 regression test).
</objective>
