<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cron tick → scheduler.ts sweep | every-minute cron triggers concurrent `runRitualSweep` / `ritualResponseWindowSweep` invocations; the same row may be observed by two ticks under the JS-clock collision window |
| cron tick → idempotency.ts | `tryFireRitualAtomic` is the chokepoint where ms-resolution `Date.now()` collisions could let two ticks both pass the WHERE predicate |
| sweep transaction → ritual_fire_events table | mid-transaction throw could leave consumedAt set without the paired audit-log rows under the pre-fix code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-42-01 | T (Tampering — data integrity) | `tryFireRitualAtomic` SET clause + WHERE predicate | mitigate | Use `sql\`now()\`` for both — postgres-clock advances strictly monotonically per-tx; second invocation's WHERE re-evaluation always fails (RACE-01 D-42-02) |
| T-42-02 | T (Tampering — data integrity) | `ritualResponseWindowSweep` paired-insert + skip_count | mitigate | Wrap atomic-consume + 2 INSERTs + skip_count UPDATE in per-row `db.transaction` — mid-row throw rolls back all four writes (RACE-02 D-42-04) |
| T-42-03 | D (Denial of Service via sweep abort) | per-row transaction failure aborting the whole sweep | mitigate | Keep the outer per-row try/catch OUTSIDE the transaction (D-42-05); single bad row's transaction failure logs + continues, sweep still processes remaining rows |
| T-42-04 | I (Information disclosure / audit-log gap) | `ritual_fire_events` table | mitigate | Per-row transaction guarantees either both audit rows + skip_count exist, or NONE exist — no half-state where the bot moved on but the audit log is missing |
</threat_model>
