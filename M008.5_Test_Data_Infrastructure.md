# v2.3 — Test Data Infrastructure *(standalone milestone, single phase, pre-M009 enabler)*

**Slash command:** `/gsd-new-milestone "v2.3 Test Data Infrastructure"`

**Positioning:** standalone milestone, not folded into M009. Reason — the primed-fixture pipeline is reusable infrastructure that every subsequent milestone (M009–M014) will consume. Keeping it independent avoids coupling and avoids each future milestone re-adding ad-hoc synthesis code.

## Goal

Build the test-data pipeline that lets any milestone be validated **immediately** — without waiting real calendar time for cron-generated data to accumulate. The pipeline pulls an **organic base** from the live Proxmox production server (192.168.1.50) and **synthesizes the delta** needed to reach the volume each milestone requires. Organic anchors realism; synthetic fills cadence. Together they give a hybrid fixture that's indistinguishable from real multi-week usage for validation purposes.

## Why this is a prerequisite

Every remaining milestone (M009–M014) currently has a real-time data-accumulation gate:

- **M009** ritual infrastructure requires *"After 2 weeks of real use"* for weekly review validation, and *"At least one month of real daily use"* before M010.
- **M008 today** is paused at 2/7 episodic summaries with an ETA of ~2026-04-24 before M009 planning can safely start.
- **M010+** profile inference layers will each add their own N-days-or-entries gates.

If every milestone adds a multi-week calendar wait, the remaining 6 milestones take ~6 months of wall-clock time — not 6 months of engineering. That's the anti-pattern this milestone eliminates. Code correctness must be verifiable on demand.

## Target deliverables

### 1. Organic data fetch script

- `scripts/fetch-prod-data.ts` — SSH to Proxmox (`192.168.1.50`) and dump relevant tables from the live `chris` Postgres:
  - `pensieve_entries` — **`source='telegram'` only**. Synced sources (immich/gmail/gdrive) are excluded from the snapshot; they don't drive consolidation (per M008.1) and their volume would bloat fixtures without adding test signal. If a future milestone specifically exercises ambient retrieval over synced sources, add them then.
  - `pensieve_embeddings` (joined by pensieve id, scoped to the `source='telegram'` ids above)
  - `episodic_summaries`
  - `decisions` + `decision_events` + `decision_capture_state`
  - `contradictions`
  - `proactive_state`
  - `memories` (M006 long-term memory rows)
- Output: `tests/fixtures/prod-snapshot/<ISO8601>/{pensieve,episodic,decisions,...}.json` — one JSON-lines file per table, with stable sort order for diffability.
- Gitignored — these are per-developer local fixtures, not committed (single-user system so there's no multi-user PII concern, but they're large and volatile).
- Uses existing SSH pattern (documented in `feedback_live_server_access.md`). No new auth.

### 2. Synthetic delta generator

- `scripts/synthesize-delta.ts --organic tests/fixtures/prod-snapshot/<stamp> --target-days 14 --seed 42` — takes an organic snapshot and extends it with synthetic day-bounded content to hit a target volume.
- **Pensieve synthesis**: generate N synthetic telegram-source pensieve entries per synthetic day, matching Greg's real conversational style sampled from the organic base. Style-transfer done via a single Haiku call per day using the organic entries as few-shot examples (stochastic sampling; seed-controlled for reproducibility). Entries distributed across realistic hours (UTC+1/+2 per season).
- **Episodic synthesis via real engine**: run the **actual** `runConsolidate(date)` from `src/episodic/consolidate.ts` against the synthetic days. This produces real Sonnet output over synthetic input — authentic consolidation behavior, controlled cadence. Costs ~$0.02/day of real Anthropic spend but produces ground-truth realistic summaries.
- **Decisions synthesis**: generate synthetic decision rows with realistic `resolve_by` spread; leave resolution state empty unless the milestone under test specifically requires resolved decisions (in which case synthesize resolution replies + run the real `handleResolution`).
- **Contradictions synthesis**: deterministic — insert N pre-written contradiction pairs with known `confidence >= 0.75`.
- **Wellbeing snapshots synthesis** (for M009): generate N synthetic rows with realistic 1–5 distributions.
- Output: a new layer `tests/fixtures/primed/<milestone>-<N>days/` containing the fused organic+synthetic dataset ready for test import.

### 3. Test harness integration

- `src/__tests__/fixtures/load-primed.ts` — loader utility that seeds the Docker Postgres test DB from a primed fixture directory. One function: `await loadPrimedFixture('m009-14days')`. Idempotent; clears target tables in FK-safe order first.
- Integrates with existing Vitest `fileParallelism: false` + Docker Postgres pattern. No new infrastructure.
- Used by future M009+ test suites instead of `vi.setSystemTime` + fully-mocked pensieve seeds.

### 4. Freshness + reproducibility guardrails

- `tests/fixtures/prod-snapshot/LATEST` symlink auto-updated by the fetch script.
- **Freshness policy — 1 day**: if `LATEST` is older than 24h, the test harness (and `synthesize-delta.ts` / `regenerate-primed.ts`) **auto-refresh** the snapshot by invoking `fetch-prod-data.ts` before proceeding. No warnings, no opt-out flag for normal runs — stale data is silently replaced. (Offline override: `--no-refresh` skips the fetch and uses whatever `LATEST` points at. For sandbox/air-gapped workflows only.)
- Seed-based synthetic generation means re-running `synthesize-delta.ts --seed 42` against the same organic base produces identical output (modulo Anthropic sampling for episodic — which is captured in a VCR-style fixture snapshot on first run and replayed thereafter).
- `scripts/regenerate-primed.ts --milestone m009 --force` — refreshes everything from scratch (fetch + synthesize + cache). Used before each milestone kickoff.

### 5. Documentation

- Brief `TESTING.md` update explaining the organic+synthetic pattern, how to use `loadPrimedFixture`, when to refresh snapshots.
- New project convention: *"no milestone may gate on real calendar time for data accumulation — use the primed fixture pipeline."*

## Acceptance

1. `npx tsx scripts/fetch-prod-data.ts` pulls current production data into `tests/fixtures/prod-snapshot/<stamp>/` (exit 0, snapshot validates against schema).
2. `npx tsx scripts/synthesize-delta.ts --organic <stamp> --target-days 14 --seed 42` produces a fused fixture with ≥ 14 episodic summaries, ≥ 200 pensieve entries, and the delta is reproducible under re-run.
3. `loadPrimedFixture('m008-14days')` seeds the Docker test DB cleanly; a sanity-check vitest passes assertions against the fused data (e.g. "7 days of summaries exist between 2026-04-15 and 2026-04-28").
4. Re-running the fetch + synthesize pipeline after a new real cron fire results in a diff-able primed fixture update (organic base grew, synthetic delta shifted to preserve target volume).
5. M008's own pause-gate (2/7 summaries) closes immediately: the primed fixture produces 7 summaries on demand. M009 planning can start today.

## Out of scope

- Anonymization / PII scrubbing: Greg is the sole user; his own prod data is the only data. If the project ever adds a second user, revisit.
- Cross-environment data migration (prod→staging): no staging environment exists.
- Synthetic *image* generation for Immich-path tests: defer until M009+ exercises vision paths.
- Voice-transcript synthesis: defer until voice-upload lands.

## Relationship to subsequent milestones

- **M009** uses this to validate the 14-day synthetic fixture assertion without waiting 14 real days — and to run the "one month of real daily use" pause-gate as "one month of primed fixture + 1 week of real confirmation" (real-daily confirmation stays, but it shrinks from blocking to sanity-checking).
- **M010+** profile-inference layers use this to pre-stage realistic multi-dimensional data before each milestone's tests, rather than adding per-milestone ad-hoc synthesis code.
- v2.3 is the *last* piece of infrastructure before the soul system; everything downstream benefits.
